<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| webhook caller -> `HandleWebhook` | Untrusted external payload enters the alert ingestion path. |
| `HandleWebhook` -> PostgreSQL `alerts` row | Server-generated trace truth becomes persistent state. |
| `HandleWebhook` -> Redis stream | Correlation data crosses a process boundary through Redis `XAdd`. |
| sync request path -> async notification goroutine | Trace context must survive after HTTP handling returns. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-14-01 | S | `internal/handlers/webhook.go` request ingest | mitigate | Mint `trace_id` server-side after datasource auth passes; ignore caller-supplied trace fields per D-01/D-03. |
| T-14-02 | T | `internal/models/alert.go` persisted trace field | mitigate | Persist trace only from handler-generated value and keep `alert_id`/`fingerprint` generation unchanged so business identifiers are not overwritten. |
| T-14-03 | R | dedup branch in `HandleWebhook` | mitigate | Emit dedup-stage trace context with `trace_id`, `fingerprint`, and `existing_alert_id` so repeated webhooks cannot silently disappear from troubleshooting history. |
| T-14-04 | I | webhook lifecycle logging | mitigate | Do not add raw `alert.Raw` or full request bodies to new trace fields/logs; restrict propagation to stable identifiers and bounded stage context. |
| T-14-05 | A | Redis handoff in `publishToRedis` | mitigate | Carry `trace_id` into Redis payloads so later stages can correlate even if request-local state is gone. |
</threat_model>
