<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| route match result -> async send loop | Internal alert/channel tuples cross from request work into a background goroutine window. |
| `WebhookHandler` -> `notifier.SendToChannel` | The handler delegates to channel-specific network senders that can fail transiently or terminally. |
| sender errors -> retry classifier | Unstructured downstream errors are interpreted into retry decisions. |
| attempt logs -> operator diagnosis | Runtime logs become the only allowed terminal evidence in this phase. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-15-01 | T | `internal/handlers/webhook.go` retry loop | mitigate | Retry only after notifier classification says the failure is transient send-stage, per D-01/D-02; keep datasource/render/config failures out of the loop. |
| T-15-02 | D | async goroutine retry window | mitigate | Use a small fixed retry count and short delay so retries remain bounded and cannot turn one webhook into long-lived background work, per D-09. |
| T-15-03 | R | attempt-level diagnosis | mitigate | Log every attempt with `trace_id`, `alert_id`, `channel_id`, `attempt`, `max_attempts`, and `error` so operators can prove whether retries happened. |
| T-15-04 | S | retry policy ownership | mitigate | Keep one shared policy for all channel types per D-05; do not let channel-specific ad hoc branches silently diverge. |
| T-15-05 | I | attempt logs | accept | Existing context fields like `channel_name`, `channel_type`, or `source` may remain if already logged, but new fields should stay within the minimum contract from D-07/D-08 and avoid raw payload leakage. |
</threat_model>
