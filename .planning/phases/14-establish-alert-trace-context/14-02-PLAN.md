<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| webhook caller -> lifecycle logs | Untrusted request data could try to influence correlation output. |
| PostgreSQL alert state -> lifecycle logs | Persisted alert fields are echoed into logs and must avoid leaking raw payloads. |
| Redis `XAdd` result -> operator troubleshooting | Publish outcomes cross from runtime internals into diagnosable operator-visible evidence. |
| route match / notification entry -> logs | Operational logs become the primary evidence path for later reliability work. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-14-06 | S | lifecycle logging helper in `internal/handlers/webhook.go` | mitigate | Always source `trace_id` from server-generated alert/request context and never from inbound headers/body when emitting lifecycle logs. |
| T-14-07 | I | ingest/persist/dedup logs | mitigate | Restrict new observability fields to identifiers and bounded error context; do not log full raw payload JSON or `alert.Raw`. |
| T-14-08 | R | Redis publish branch | mitigate | Capture both success and failure with `trace_id`, stream name, and returned message id/error so Redis handoff cannot fail silently. |
| T-14-09 | R | route/dedup lifecycle evidence | mitigate | Emit explicit stages for route miss/hit and dedup decisions so operators can prove whether the alert entered routing/notification. |
| T-14-10 | D | backend logging scope | accept | Global logging standardization is deferred to Phase 16; Phase 14 intentionally limits changes to the webhook path to avoid destabilizing unrelated flows. |
</threat_model>
