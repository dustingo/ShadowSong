✅ 已完成功能

  • 告警 CRUD + 状态流转 (firing/acked/silenced)
  • 告警去重 (fingerprint)
  • 路由规则 (级别+来源→渠道匹配)
  • 静默规则 (手动创建)
  • 多渠道投递 (Webhook/钉钉/企微)
  • 投递记录 + 重试机制
  • Webhook 模板渲染
  • 大盘统计 (趋势图/级别分布)
  • 用户认证 + 角色权限

⚠️ 半成品 / 问题

  • 值班管理 — 数据录入完整，但与告警流程断路
  • 静默规则 — 仅手动创建，不支持 label pattern
  • 通知模板 — 仅 Webhook 支持，钉钉/企微硬编码
  • 投递记录 — 后端有数据，前端无展示页面
  • 告警确认 — 仅状态切换，无超时/升级机制

🎯 建议功能优先级矩阵

功能 解决什么痛点 实现复杂度 用户感知度 建议优先级
告警聚合/降噪 告警风暴时列表刷屏，无法快速定位 中 极高 P0 必做
确认与升级机制 告警 acked 后无人跟进，不了了之 中 高 P1 重要
投递可观测性 通知到底发没发出去？不知道 低 高 P1 重要
通知模板统一 钉钉/企微消息格式不可定制 中 中 P2 改善
操作审计日志 谁操作了什么，无据可查 低 中 P2 改善
告警依赖/关联 根因和下游混在一起 高 中 P2 改善
SLA 响应时间 MTTA/MTTR 无量化 低 低 P2 改善
值班管理 与现有渠道模型冲突，孤岛功能 砍掉

🔗 功能依赖关系

告警聚合 → 升级机制 聚合后才能对"一组告警"做升级，而非逐条升级
告警聚合 → 通知模板 聚合后消息格式需展示"3条同类告警"而非单条
升级机制 → 通知模板 升级通知需要不同的消息模板（紧急/催办）
投递可观测 → 升级机制 升级需要知道"上一次通知是否成功送达"
