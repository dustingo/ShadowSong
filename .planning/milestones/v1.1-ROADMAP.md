# Milestone v1.1: Enterprise Access Control

**Status:** ✅ SHIPPED 2026-04-15
**Phases:** 5-9
**Total Plans:** 15

## Overview

这个里程碑围绕“谁能看、谁能改、谁能管人、关键动作如何留痕”四条主线推进。先统一现有角色模型并把权限判定收口到后端，再完成用户管理边界、账号控制和配置接口加固，补上前端显隐与验证，最后通过一个小型 cleanup phase 收口文档漂移、测试噪音和残余实现细节，确保系统从“登录即可大部分可改”演进到“最小权限、服务端强制控制、关键操作可审计”的企业级基线。

## Phases

### Phase 5: Normalize Role Model

**Goal**: 在不破坏现有登录链路的前提下，统一 `admin`、`operator`、`viewer` 角色语义，并建立后端可复用的权限判定基线。  
**Depends on**: Phase 4  
**Plans**: 3 plans

Plans:

- [x] 05-01-PLAN.md — Normalize role constants, validation, and token/session claims around existing role names
- [x] 05-02-PLAN.md — Audit and harden existing user/bootstrap data compatibility without role renaming
- [x] 05-03-PLAN.md — Introduce reusable authorization helpers and backend permission matrix tests

**Details:**  
统一了固定角色真源、JWT principal 和 capability matrix 基线，为后续用户管理、配置鉴权和前端权限感知提供了共享权限契约。

### Phase 6: Secure User Administration

**Goal**: 让用户管理明确区分“管理员管理用户”和“普通用户维护个人资料”，杜绝越权改其它用户或自提权，并补齐账号禁用与强制改密控制。  
**Depends on**: Phase 5  
**Plans**: 3 plans

Plans:

- [x] 06-01-PLAN.md — Add account state and session invalidation foundations for disabled and forced-reset users
- [x] 06-02-PLAN.md — Split admin-managed user operations from self-service profile/password flows
- [x] 06-03-PLAN.md — Expose the minimal frontend users/profile surfaces for Phase 6

**Details:**  
完成了管理员管人与普通用户自助边界拆分，落地了禁用账号、强制改密和旧会话失效控制，并补齐 `/users` / `/profile` 前端入口。

### Phase 7: Lock Down Protected Operations

**Goal**: 对系统配置和运维操作接口补齐角色校验，并为关键安全动作建立后端审计，确保不同角色只能执行其职责范围内的动作。  
**Depends on**: Phase 6  
**Plans**: 3 plans

Plans:

- [x] 07-01-PLAN.md — Apply role guards to alert action endpoints and configuration route groups
- [x] 07-02-PLAN.md — Refactor handler-level authorization and add audit logging for critical account/config actions
- [x] 07-03-PLAN.md — Add endpoint-level regression coverage for role allow/deny and audit cases

**Details:**  
配置写接口与告警动作已完成服务端强制权限收口，未授权访问边界稳定为 `401/403`，并为关键安全动作写入后端持久化审计日志。

### Phase 8: Ship Permission-Aware UI And Verification

**Goal**: 让前端菜单、页面、按钮和提示与后端权限边界一致，并交付完整的角色矩阵验证与文档说明。  
**Depends on**: Phase 7  
**Plans**: 3 plans

Plans:

- [x] 08-01-PLAN.md — Add frontend role-aware menu, route, and action visibility controls
- [x] 08-02-PLAN.md — Normalize forbidden-state UX and role-specific page behavior
- [x] 08-03-PLAN.md — Document the access model and add end-to-end verification evidence

**Details:**  
前端菜单、路由、按钮和只读提示已按权限感知收口，并补齐了角色矩阵测试、验证文档和关键安全路径验证。

### Phase 9: Close v1.1 Milestone Tech Debt

**Goal**: 收口 v1.1 审计中发现的非阻塞技术债，确保项目状态文档、前端验证输出和残余权限判断实现与已交付的权限体系保持一致。  
**Depends on**: Phase 8  
**Plans**: 3 plans

Plans:

- [x] 09-01-PLAN.md — Update project-state documentation to reflect Phase 8 and milestone-complete status
- [x] 09-02-PLAN.md — Reduce frontend permission-test warning noise without regressing coverage
- [x] 09-03-PLAN.md — Replace residual raw role checks with shared authorization helpers

**Details:**  
补齐了 `PROJECT` 真相文档、清理了前端测试 warning 噪音、退役了未使用的 `RequireRole` seam，并加固了 VER-04 相关回归验证，令里程碑达到可归档状态。

## Milestone Summary

**Decimal Phases:**

- None

**Key Decisions:**

- 保持 `admin`、`operator`、`viewer` 三档固定角色，不做角色重命名迁移
- 权限控制以后端拒绝未授权请求作为最终防线，前端负责可见性与体验收口
- 配置类写权限仅授予 `admin`，`operator` 保留告警处理能力
- 将审计日志、账号禁用和强制改密纳入本里程碑

**Issues Resolved:**

- 统一了后端角色真源、principal 和 capability matrix
- 收紧了用户管理越权路径并补齐账号控制安全边界
- 解决了“任意登录用户可改配置”与关键动作无审计的问题
- 让前端菜单、页面、按钮与后端权限边界一致
- 收口了 `PROJECT` 文档漂移、前端测试 warning 噪音和残余 authz seam 漂移

**Issues Deferred:**

- 是否引入更细粒度权限点或授权继承模型
- 是否接入 SSO / LDAP / OAuth 企业身份源
- 是否为关键管理操作引入审批流

**Technical Debt Incurred:**

- `frontend/src/App.test.tsx` 仍保留局部 `console.warn` spy，用于稳定测试输出；不是 blocker，但后续仍可继续优化
