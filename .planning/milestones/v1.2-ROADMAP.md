# Milestone v1.2: Alert Pipeline Hardening

**Status:** ✅ SHIPPED 2026-04-21
**Phases:** 10-13
**Total Plans:** 8

## Overview

v1.2 围绕“实时告警访问面是否安全、前端质量门禁是否恢复、通知链路是否可可靠排障”三条主线推进。先收紧 WebSocket 订阅入口的服务端鉴权与来源校验，再修复前端 lint 红线并恢复本地质量基线，随后接入 CI 把测试、lint 与 build 变成自动门禁，最后补齐 webhook 异步通知的 panic 防护与失败追踪，确保系统在现有非 AI 基线之上继续稳定演进。

## Milestone Summary

**Key Decisions:**

- 继续维持现有 Go + Gin + GORM + PostgreSQL + Redis + React + Vite 技术栈，不通过技术迁移解决当前里程碑问题
- 先完成 WebSocket 访问面收口和前端质量基线恢复，再把这些本地已验证命令接成自动化门禁
- 通知链路可靠性优先做最小侵入的 panic 防护、失败上下文和可测试边界，不扩展到新异步基础设施
- 文档与命名只收口当前会误导工程真相的低风险入口，不在 v1.2 做 Go module 路径全量迁移

**Issues Resolved:**

- 实时告警 WebSocket 访问面此前可被匿名或非法来源直接尝试订阅
- 前端 lint 红线和高噪音 warning 持续污染工程基线
- 仓库缺少自动质量门禁，回归风险只能靠人工发现
- webhook 异步通知链路在 panic 和发送失败场景下缺乏稳定排障上下文

**Issues Deferred:**

- Go module / import 路径中更深层的历史 `ai-alert-system` 命名残留
- 更复杂的 CI 矩阵、跨平台 runner、bundle 体积治理
- 通知重试队列、持久化发送记录、集中可观测性平台
- 下一里程碑需求与 roadmap 尚未定义，待 `/gsd-new-milestone`

**Technical Debt Incurred:**

- 通知可靠性增强仍以应用内日志和本地测试为主，尚未引入持久化或集中观测体系
- README/包名已对齐，但历史规格文档和少量测试名仍保留旧时代背景痕迹
- 当前未单独生成 v1.2 milestone audit 文档，本次归档基于 phase-level verification 与全量测试完成
