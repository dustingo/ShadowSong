## 开发工具
   语言：go

## 一、核心数据结构

### 统一告警结构体
所有数据源的告警经过 Template 处理后，统一转换为以下结构：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| alert_id | string | 是 | 系统生成 UUID |
| source | string | 是 | 数据源名称 |
| alert_name | string | 是 | 告警名称 |
| severity | string | 是 | 统一级别 P0/P1/P2/P3 |
| message | string | 是 | 告警描述 |
| labels | map[string]string | 否 | 所有维度字段，内容由数据源 Template 决定 |
| fingerprint | string | 系统生成 | 去重指纹 |
| trigger_time | timestamp | 是 | 告警触发时间，UTC |
| received_at | timestamp | 系统生成 | 系统接收时间 |
| status | string | 系统生成 | pending/firing/acked/silenced/resolved |
| raw | json | 系统生成 | 原始告警数据完整保留 |

**关键规则：**
- 不存在固定的 `host` 字段，主机信息作为 labels 的一个 key 存储（如 `labels.host`）
- `alert_name`、`severity`、`message` 是仅有的三个必填业务字段，其余维度信息全部放入 labels
- labels 的 key 和 value 均为字符串，value 不支持嵌套结构
- Template 中可以将原始数据的任意字段映射为 labels 中的任意 key

## 二、数据源管理

### 功能
- 每个数据源对应一个唯一的 Webhook 接收地址：`/webhook/{source_name}`
- 通过 Go Template 将原始告警 JSON 转换为统一结构体
- 在数据源配置中指定分组维度（`group_by_labels`）
- 支持在前端在线编辑模板并实时测试

### 数据源配置项

| 配置项 | 类型 | 说明 |
|--------|------|------|
| name | string | 唯一标识，用于 webhook 路径，创建后不可修改 |
| display_name | string | 显示名称 |
| template | text | Go Template 内容 |
| group_by_labels | []string | 分组维度，从 labels 的 key 中选择 |
| enabled | bool | 是否启用 |

### Template 可用的自定义函数

| 函数 | 说明 |
|------|------|
| `uuidv4` | 生成 UUID |
| `mapValue val k1 v1 k2 v2...` | 值映射，未匹配返回原值 |
| `timeFormat val inputLayout outputLayout` | 时间格式转换，建议统一输出 RFC3339 |
| `defaultVal val default` | 字段为空时返回默认值 |
| `regexExtract pattern val` | 正则提取第一个捕获组 |
| `toLower` / `toUpper` | 大小写转换 |

### 关键规则
- `name` 创建后不允许修改，因为外部系统已配置好 webhook 地址
- 数据源禁用后，对应 webhook 返回 403，不接收告警
- 模板必须能渲染出 `alert_name`、`severity`、`message` 三个必填字段，否则验证不通过，不允许保存
- severity 必须是 P0/P1/P2/P3 之一，Template 中应使用 `mapValue` 将原始值映射为标准值
- `group_by_labels` 为空时，默认使用 `alert_name` 单独作为分组维度
- `group_by_labels` 中指定的 key 在 labels 中不存在时，该 key 的值视为空字符串参与分组计算，不报错
- 模板渲染失败时，系统生成降级告警（severity=P1，message=原始数据前500字节，labels为空）继续流转，并写入死信日志

### 模板测试功能
- 前端提供模板编辑框（代码高亮）和样本输入框
- 点击测试后调用后端验证接口，返回解析后的统一结构体或具体错误信息
- 新增/编辑数据源保存前必须通过一次测试验证

## 三、告警接入流程

```
POST /webhook/{source_name}
        ↓
数据源是否启用？ → 否 → 返回 403
        ↓ 是
Template 渲染 → 失败 → 生成降级告警 + 写死信日志
        ↓ 成功
计算 fingerprint
        ↓
写入 Redis Stream（alerts:normalized）
        ↓
返回 200 + alert_id
```

## 四、告警指纹与去重

### 指纹计算规则
```
fingerprint = SHA256(
    source + "|" +
    alert_name + "|" +
    按字母序排列的 group_by_labels 键值对
)
```

示例：
- 数据源 `group_by_labels = ["host", "zone"]`
- 某条告警 labels = `{"host": "game-server-01", "zone": "east-1", "env": "prod"}`
- 参与指纹计算的 labels 只取 group_by_labels 中指定的：`host=game-server-01, zone=east-1`
- 最终指纹 = `SHA256("prometheus|HighMemory|host=game-server-01|zone=east-1")`

**关键规则：**
- 指纹计算维度与分组维度保持完全一致，去重和聚合使用同一套 key，不会出现矛盾
- 相同指纹的告警在去重 TTL 内只允许一条进入后续流程
- 去重 TTL 默认 5 分钟，可在系统配置中调整
- 被去重的告警不进入聚合和 AI 流程，但仍然入库，状态标记为 `deduplicated`，同时将对应活跃告警的触发计数 +1
- 每条原始告警都入库，去重只影响后续处理流程，不影响数据完整性

## 五、静默匹配

### 匹配规则
静默规则的匹配条件支持以下字段，**所有配置的条件必须同时满足**才命中：

| 条件字段 | 匹配方式 | 说明 |
|----------|----------|------|
| source | 精确匹配 | 为空则匹配所有来源 |
| alert_name | 正则匹配 | 为空则匹配所有告警名 |
| severity | 多选匹配 | 为空则匹配所有级别 |
| labels | key-value 正则匹配 | 支持配置多个 label 条件，每个条件 value 支持正则 |

**关键规则：**
- 静默规则有明确的生效时间范围（starts_at ~ ends_at），到期自动失效
- 命中静默的告警状态标记为 `silenced`，不进入聚合窗口，不触发 AI，不发送通知
- 静默规则缓存刷新间隔 30 秒
- 被静默的告警在前端可查询，但不在大盘活跃列表中展示
- 取消静默后，历史已静默告警状态不回滚，新触发的同类告警恢复正常处理

## 六、窗口聚合

### 分组 key 计算
```
group_key = source + "|" + alert_name + "|" + 按字母序排列的 group_by_labels 键值对
```

与指纹计算逻辑一致，保证同一指纹的告警必然在同一分组内。

### 窗口触发条件（满足任一即触发）
1. 该组最后一条告警到达后，超过 **10 秒**没有新的同组告警（静默期）
2. 该组第一条告警到达后，已等待超过 **30 秒**（最大等待兜底）

**关键规则：**
- 静默期和最大等待时间可在系统配置中调整
- 单组最大告警数量上限 100 条，超出部分丢弃，防止告警风暴撑爆内存
- 窗口触发后该组关闭，新来的同组告警开启新窗口
- 窗口触发后，将聚合组（AggregatedAlert）送入 AI 处理队列

### 完整处理流水线
```
Redis Stream 消费
        ↓
去重检查 → 命中 → 触发计数+1，入库，结束
        ↓ 未命中
静默匹配 → 命中 → 标记 silenced，入库，结束
        ↓ 未命中
窗口聚合（按 group_key 分组，Session Window）
        ↓ 窗口触发
AI 处理
        ↓
通知路由分发
```

## 七、AI 自动处理

### 功能
对聚合后的告警组调用大模型，输出：
- 一句话摘要
- 根因推断
- 综合评估后的最终级别
- 处置建议步骤列表
- 是否为已知自愈型告警
- 标签列表（如"内存泄漏"、"网络抖动"）

### AI 上下文
每次调用 AI 时，同时传入：
- 当前聚合告警组的完整信息
- 同类告警（同 source + alert_name）最近 10 条已处置记录（status=acked），包含处置备注

### 关键规则
- AI 调用超时时间 15 秒
- 超时或调用失败时降级处理：跳过 AI，使用聚合组中最高 severity 作为最终级别，第一条告警的 message 作为摘要，继续流转推送，不阻塞
- AI 输出的 severity 不得低于聚合组原始最高 severity 两个档位（如原始最高 P1，AI 最低只能给 P2）
- AI 处理的完整输入和输出均入库，用于复盘和 Prompt 优化
- AI 对告警数据只有读权限，不执行任何写操作

## 八、通知路由

### 路由规则匹配
路由规则支持以下匹配条件，**所有配置的条件同时满足**才命中：

| 条件字段 | 匹配方式 |
|----------|----------|
| severity | 多选（如 P0、P1） |
| source | 多选 |
| labels | key-value 正则匹配，支持多个条件 |

**关键规则：**
- 路由规则按 priority 排序，数值越小优先级越高
- 匹配到第一条规则后停止（first-match 模式）
- 无规则命中时：P0/P1 推送到系统默认渠道，P2/P3 仅入库
- 路由规则支持配置通知时间段（如仅 09:00-18:00），时段外命中的告警仅入库
- 当前值班人员的渠道**叠加**到路由匹配结果上，不替换原有渠道
- 推送失败时重试 3 次，间隔 5 秒，仍失败则记录日志不再重试

## 九、推送渠道

### 支持的渠道类型

**飞书：**
- 配置项：webhook_url、secret（用于签名校验）
- 消息格式：富文本 Interactive Card
- 消息内容：告警级别、来源、触发时间、AI摘要、根因推断、建议步骤、关联告警数量
- 操作按钮：「确认处理」「查看详情」
- 按钮链接：指向系统告警详情页和一键确认接口

**钉钉：**
- 配置项：webhook_url、secret（用于加签）
- 消息格式：ActionCard
- 消息内容：同飞书
- 操作按钮：「确认处理」「查看详情」

**企业微信：**
- 配置项：webhook_url
- 消息格式：Markdown
- 消息内容：同飞书，末尾附确认和详情链接（不支持按钮）

**自定义 Webhook：**
- 配置项：url、method（默认POST）、headers（支持自定义，如 Authorization）、body_template（Go Template，为空时发送标准 JSON）
- 标准 JSON 包含：alert_id、group_id、severity、summary、root_cause、suggestions、labels、alert_count、trigger_time、ack_url、detail_url
- 超时时间：10 秒

### 消息级别视觉规范

| 级别 | Emoji | 含义 |
|------|-------|------|
| P0 | 🔴 | 核心服务故障，立即处理 |
| P1 | 🟠 | 重要服务异常，尽快处理 |
| P2 | 🟡 | 一般告警，关注 |
| P3 | 🔵 | 低优先级，了解即可 |

### 关键规则
- 敏感配置（webhook_url、secret）在前端展示时脱敏
- 测试发送功能：发送一条固定格式的测试消息，验证渠道配置是否正确
- 渠道禁用后，路由到该渠道的告警跳过该渠道，不报错

## 十、告警确认（Ack）

### 功能
- 运维人员确认告警，填写处置备注
- 支持在系统前端确认
- 支持通过飞书/钉钉消息中的「确认处理」按钮直接确认（无需登录系统）

### 关键规则
- 确认后状态变为 `acked`，从大盘活跃列表移除
- 一条告警只能被确认一次，重复确认返回提示
- 通过消息按钮确认时使用带签名的一次性 token，有效期 24 小时，防止链接被滥用
- 确认信息（操作人、时间、备注）永久保留

## 十二、值班管理

### 功能
- 按时间段配置值班人员及对应的通知渠道
- 值班期间，该渠道叠加到所有 P0/P1 告警的推送目标中

- 值班渠道属于额外叠加，不替换路由规则配置的渠道
- 当前值班人查询结果缓存 60 秒

## 十三、告警状态流转

```
进入系统
    ↓
pending
    ↓ 推送成功
firing ──→ silenced（命中静默规则）
    │
    ├──→ acked（人工确认）
    │
    └──→ resolved（数据源发来 resolved 事件）
```

**关键规则：**
- `resolved` 事件：部分数据源（如 Prometheus Alertmanager）会主动发送恢复通知，Template 中需映射 `event_type` 字段（firing/resolved），resolved 事件触发时将对应活跃告警状态改为 resolved，并发送恢复通知
- 不支持 resolved 事件的数据源，告警状态不会自动变为 resolved，需人工确认

## 十五、前端页面
### 风格
- 参考https://ui-ux-pro-max-skill.nextlevelbuilder.io/的风格、配色设计，注意是浅色系的
### 告警大盘
- 顶部统计卡片：P0/P1/P2/P3 各级别当前活跃数量
- WebSocket 实时推送新告警，断线自动重连（间隔 3 秒）
- 活跃告警列表：按 severity（P0优先）+ trigger_time（新的优先）排序
- P0 告警顶部固定并红色高亮
- 大盘只展示 status=firing 的告警
- 最近 24 小时告警趋势折线图
- 列表支持快速确认操作

### 告警管理
- 全量告警列表，支持按级别、来源、状态、时间、labels 筛选
- 点击展开：原始告警详情 + AI 分析结果（摘要、根因、建议、标签）
- 操作：确认（填写备注）、快速静默（预填匹配条件）
- 告警触发计数展示（去重期间同一问题触发了多少次）

### 数据源管理
- 列表：名称、webhook 地址、分组维度、状态、最近触发时间
- 新增/编辑：
  - 名称、显示名称
  - Template 编辑框（代码高亮）
  - group_by_labels 配置（输入 label key，支持多个）
  - 测试区：输入原始 JSON 样本 → 点击测试 → 展示解析结果或错误详情
- 启用/禁用开关
- 禁用时提示："禁用后 /webhook/{name} 将停止接收告警"

### 推送渠道管理
- 列表：名称、类型、状态
- 新增/编辑：按渠道类型显示对应配置表单
- 敏感信息脱敏展示
- 测试发送按钮

### 路由规则
- 列表展示所有规则，支持拖拽排序（调整优先级）
- 新增/编辑：
  - 匹配条件：severity 多选、source 多选、labels 条件（key + 正则 value，可添加多条）
  - 目标渠道：多选
  - 通知时间段（可选）
- 优先级以数字展示，拖拽后自动重新编号

### 静默管理
- 活跃规则列表 / 历史规则列表（Tab 切换）
- 新增静默：填写名称、备注、匹配条件（支持 label 条件）、时间范围
- 快捷时长按钮：1小时 / 4小时 / 今天结束
- 支持提前取消

### 值班管理
- 日历视图展示排班
- 新增排班：选择用户、时间段、通知渠道

### AI 查询（新增）

**告警智能问答：**
- 页面内嵌对话框，自然语言提问
- 支持查询类型：统计、趋势、历史处置记录、跨服务对比
- AI 只读，不执行任何写操作
- 每次对话保留最近 10 轮上下文
- 涉及数字结果时优先以图表展示
- 每次回答末尾注明数据范围（如"基于最近 30 天告警数据"）

**静默规则推荐：**
- AI 每天定时分析历史数据
- 触发推荐条件（同时满足）：
  - 同一指纹告警最近 7 天出现 ≥ 5 次
  - 历史自愈率（未经人工处置自动恢复）≥ 80%
  - 触发时间有明显规律
- 推荐以卡片形式展示，预填好匹配条件和时间范围
- 运维人员可修改后一键采纳，或忽略
- 忽略的推荐 7 天内不重复出现

**处置建议查询：**
- 告警详情页内嵌「问 AI」按钮
- AI 检索同类告警（同 source + alert_name）最近 20 条已处置记录
- 历史记录不足 3 条时，AI 注明"历史数据较少"并给出通用建议
- 建议以有序步骤列表展示

**AI 处理日志：**
- 展示每次 AI 自动处理的输入和输出
- 运维人员可标记「准确」或「不准确」并填写反馈
- 展示近 7 天 / 近 30 天 AI 判断准确率统计
- 标记为「不准确」的记录高亮展示，便于批量审查

## 十六、数据保留策略
- 原始告警数据：保留 90 天
- AI 处理日志：保留 90 天
- 死信日志：保留 30 天
- 超过 90 天的已处置告警归档至历史表，不影响主表查询性能
